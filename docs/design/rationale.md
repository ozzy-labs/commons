# 設計思想 (Rationale)

OzzyLabs における技術選定と設計の背景を記します。

## リンター・フォーマッターの選定理由

### 1. Biome (JSON, JavaScript/TypeScript)

- **選定理由**: 圧倒的な速度と、リンター・フォーマッター・インポート整理が単一ツールで完結する利便性。
- **背景**: 以前は ESLint + Prettier を使用していたが、設定の複雑さとパフォーマンスの問題から、Rust 製の高速な Biome への移行を標準とした。

### 2. markdownlint / mdformat (Markdown)

- **選定理由**: Markdown の構造的妥当性を担保するため。
- **背景**: `mdformat` は「折り返しなし」設定を強制することで、Git の diff を見やすく保つ（1段落1行）。

### 3. yamlfmt / yamllint (YAML)

- **選定理由**: YAML はインデントに敏感であり、ミスが原因で CI が壊れることが多いため、厳格なバリデーションが必要。
- **背景**: Renovate や GitHub Actions 等で多用される YAML の品質を一定に保つ。

### 4. ShellCheck / shfmt (Shell)

- **選定理由**: シェルスクリプトは記述が容易な反面、落とし穴（クォート忘れ等）が多い。
- **背景**: `sync.sh` 等のコアツールが Bash で書かれているため、その安全性と読みやすさを担保するために必須。

### 5. Trivy / Gitleaks (Security)

- **選定理由**: 「セキュリティの左シフト（開発初期段階での検知）」。
- **背景**: シークレットの誤コミットや依存関係の脆弱性を、GitHub にプッシュする前のプリコミットフック段階で阻止する。

## ブランチ・コミット規約の設計理由

### 1. Squash Merge の強制

- **理由**: `main` ブランチの履歴を「1 PR = 1 コミット」の粒度で保つため。
- **背景**: リバートの容易さと、変更履歴（`git log`）の読みやすさを最優先する。マージコミットによるノイズを排除する。

### 2. Conventional Commits の採用

- **理由**: コミットメッセージの構造化による、リリースメモの自動生成とセマンティックバージョニング（SemVer）への対応。
- **背景**: `feat:`, `fix:` 等の接頭辞を付けることで、変更の種類を一目で判別可能にする。

### 3. main 保護ルール

- **理由**: 「直接 push」による未検証コードの混入を物理的に防ぐ。
- **背景**: たとえソロ開発であっても、PR を経由することでセルフレビューと CI による検証を強制し、品質を維持する。

## なぜ `sync.sh` という独自スクリプトなのか

### 1. 疎結合な同期

- **理由**: Git サブモジュールや npm パッケージ化は、管理オーバーヘッドが大きく、consumer 側の自由度を奪いすぎる。
- **背景**: 「推奨設定をコピーするが、必要なら pin (固定) して乖離できる」という、OzzyLabs 流の「オプショナルな標準化」を実現するための最適解。

### 2. 多様な技術スタックへの対応

- **理由**: Node.js リポジトリだけでなく、Python, Go, Rust などあらゆる言語のリポジトリで、共通の `.editorconfig` や `.gitignore` を使いたい。
- **背景**: 言語固有のパッケージマネージャーに依存しない、単独で動作する Bash スクリプトを採用。
